disjoint-set (并查集)

union-find 或 merge-find (并查集)

并查集是一种非常精巧而实用的数据结构，它主要用于处理一些不相交集合的合并问题。

一些常见的用途有求连通子图、求最小生成树的 Kruskal 算法和求最近公共祖先（Least Common Ancestors, LCA）等。

是解决动态连通性问题的一类非常高效的数据结构。

设计 API

初始化，连接两个触点，判断包含某个触点的分量，判断两个触点是否存在于同一个分量之中，以及返回所有分量的数量。

UF 算法的 API

Quick-Find

当且仅当 id[p] 等于 id[q] 时，p 和 q 是连通的，即在同一个连通分量中的所有触点在 id[] 中的值必须全部相等。

quick find 概览

LeetCode Playground

quick-find 算法：find() 操作速度很快，因为只需要访问 id[] 数组一次，但 quick-find 算法无法处理处理大型问题，因为对于每一对输入 union() 都需要扫描整个 id[] 数组。

在 quick-find 算法中，每次 find() 调用只需要访问数组一次，而归并两个分量的 union() 操作访问数组的次数在 $(N + 3)$ 到 $(2N + 1)$ 之间。

Quick-Union

quick-union 算法提高了 union() 方法速度。

使用 Tree 数据结构优化

quick-union 算法看起来比 quick-find 算法更快，因为不需要遍历整个数组

quick union 概览

quick-union 算法可以看作是 quick-find 算法的改良。因为解决了quick-find 算法中最最要的问题（ union() 操作总是线性级别的）。但是 quick-union 算法仍然存在问题，我们无法保证在所有情况下都比 quick-find 算法快。

Weighted Quick-Union

我们只需要简单的修改 quick-union 算法就能确保最糟的情况不在出现。与其在 union() 中随意的将一棵树连接到另一棵树，先在记录每棵树的大小并且总是将小的树连接到较大的树上。这个改动需要添加一个数组和一些代码来记录树中的节点数，但能大大改进算法的效率。

Tree 连接可能存在大树连到小树上的情况，为避免大树连到小树的情况，我们记录树的大小，并且总是将小的树连接到大的树。 如下图：

分析

这个算法能够处理规模较大的问题。

对于 $N$ 个触点，加权 quick-union 算法构造的森林中的任意节点的深度最多为 $lgN$。

对于加权 quick-union 算法和 $N$ 个触点，在最坏的情况下 find(), connected() 和 union() 的成本的增长的数量级为 $lgN$。

对于动态连通性问题，加权 quick-union 算法是三种算法中唯一可以解决大型实际问题的算法。

只要多写几行代码，我们所得到的程序在处理实际应用中的大型动态连通性问题时就会比简单的算法快数百万倍。
